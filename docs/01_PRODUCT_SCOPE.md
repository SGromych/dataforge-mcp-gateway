# 01. Product Scope

## Название продукта

Рабочее название:

**DataForge Semantic MCP Server**

Возможные альтернативы:

- DataForge MCP Semantic Gateway
- DataForge RMD MCP Server
- DataForge Semantic Context Server

## Что делает продукт

Продукт представляет собой отдельный MCP-сервер, задача которого — передавать в AI-агенты семантический слой из DataForge.

Под "семантическим слоем" в данном контексте понимаются:

- список доступных проектов
- список версий по проекту
- меры (`measures`)
- измерения (`dimensions`)
- полный RMD (`measures + dimensions`)
- производные представления этих сущностей в нормализованном виде

## Что продукт НЕ делает

На текущем этапе продукт **не должен**:

- подключаться к аналитическим БД
- выполнять SQL
- генерировать SQL
- проверять SQL
- заниматься text-to-SQL
- строить BI-дашборды
- вызывать внешние LLM
- хранить бизнес-данные, кроме кэша RMD
- редактировать объекты в DataForge

Иными словами, это **read-only semantic MCP adapter**.

## Зачем нужен отдельный продукт

Выделение MCP в отдельный репозиторий и продукт дает несколько преимуществ.

Во-первых, появляется четкая зона ответственности. DataForge остается системой управления семантикой и моделями, а MCP-сервер — интеграционным слоем для AI.

Во-вторых, появляется повторное использование. Один и тот же MCP-сервер можно подключать к разным агентам: Claude Desktop, ChatGPT-compatible clients, VS Code extensions, кастомным AI-агентам, orchestration layer, internal copilots.

В-третьих, упрощается конфигурирование. Вместо захардкоженного скрипта появляется продукт, который можно развернуть, настроить и сопровождать отдельно.

## Основной принцип проектирования

Сервер должен строиться вокруг следующей идеи:

**AI-агентам нужен не "сырой API DataForge", а стабильный MCP-контракт для доступа к семантическому слою.**

Следовательно, продукт должен:

- инкапсулировать DataForge API
- нормализовать его ответы
- скрывать от агентов детали транспортного уровня
- давать удобные и предсказуемые MCP tools
- управлять кэшем, ошибками и форматами ответа

## MVP-границы

Минимальный жизнеспособный продукт должен уметь:

- проверить доступность Product API
- получить список проектов
- получить список версий проекта
- получить full RMD для проекта/версии
- получить measures отдельно
- получить dimensions отдельно
- нормализовать данные
- вернуть эти данные через MCP
- кэшировать ответы
- писать аудит вызовов

## Non-goals первого релиза

Первый релиз не должен пытаться решить все сразу.

Не надо добавлять:

- multi-tenant UI
- web admin panel
- BI previews
- embeddings
- vector search
- поиск по документации
- редактирование метрик
- синхронизацию с DWH
- approval workflows
- data lineage visualization

Это можно добавить потом как roadmap, но не надо включать в первую реализацию.
